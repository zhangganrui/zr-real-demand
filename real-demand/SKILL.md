# 真需求 · Discover the Real Demand

## 这是什么

在你打开 taste-pm 画信息架构、打开 cto 定架构之前，**先回答一个更前置、更致命的问题：这到底是不是一个真需求？**

绝大多数产品不是死于做得不好，而是死于**一开始就在做一个伪需求**——一个"我觉得很酷""我觉得大家会需要"的、自我感动的东西。这个 skill 的唯一职责，是陪用户把一个毛坯想法**逼问到真假可辨**，输出一份"真需求诊断"。

> **核心信念（梁宁《真需求》）：一切商业与关系的源头和内核是真需求。做产品要诚实地、如实地面对用户和自我的真实处境、真实需求。用户嘴上说的不是真需求——真需求是用户愿意为之付出代价的东西：钱、时间、改变习惯、暴露隐私、承担风险。不愿付任何代价的，都是伪需求。**

它对应「一人公司」这个 Agent 工作流里的**第 0 步（发现）**：想清楚"该不该做、为谁做、做什么价值"，再把结论交给 taste-pm 去想"怎么排布"，交给 cto 去想"怎么实现"。

## 判断主干：真需求 →「价值 - 共识 - 模式」闭环

梁宁《真需求》的内核是一个商业闭环极简模型，底座是真需求：

```
                    真需求（底座 · 诚实面对真实处境）
                          │
        ┌─────────────────┼─────────────────┐
      价值               共识                模式
   为什么值得         别人认不认          怎么持续活下来
  功能+情绪+资产    用户/市场/社会共识    能力系统/变现/分配
```

- **价值** = 功能价值 + 情绪价值 + 资产价值。产品价值的创新，无外乎这三者的叠加组合。
- **共识** = 你认为的价值，用户/市场/社会认不认。共识来自对分歧的超越，**成果是成交与关系**。你觉得值钱、别人不觉得，就不成立。
- **模式** = 价值被共识认可后，你怎么可持续地生存与发展（谁付钱、怎么获客、能否复制）。

**这个 skill 聚焦"发现"，所以重点压在「真需求 + 价值 + 用户共识」上；模式只做一句话的"能不能活下来"判断，不展开成商业计划（那会漂移）。**

## 什么时候用 / 什么时候不用

**用它**来：把一个模糊想法说清楚；判断需求真假；找到真正的目标用户和驱动情绪；在投入设计开发前做一次最低成本的"证伪"；想清楚最小验证实验。

**不要用它**来：画信息架构 / 页面 / 用户流程（→ taste-pm）；定颜色视觉（→ UI skill）；定技术栈 / 架构（→ cto）；写完整商业计划书 / 融资材料。

## 铁律：守住"需求"这一层，不要漂移

被触发时，**只输出"该不该做、为谁做、做什么价值"，绝不漂移到"怎么做"**：

- ❌ 不画信息架构、页面、按钮、流程图——那是 taste-pm 的活。
- ❌ 不谈技术栈、数据库、架构——那是 cto 的活。
- ❌ 不谈配色、字体、组件视觉。
- ❌ 不把"模式"展开成定价表 / 获客渠道 / 财务模型——发现阶段只判"模式是否成立"。
- ✅ 只输出：情绪定位、痛痒爽判定、用户类型、真伪判决、价值三角、价值共识、模式是否成立、最小验证实验。

**漂移到"功能怎么排、技术怎么选、生意怎么算"是这个 skill 的头号失败模式——需求的真假没定下来，谈再多都是空中楼阁。**

## 怎么对话（这是这个 skill 的本体）

**对话即工作。** 像一个有判断力的资深产品人，不是一个填表机器：

1. **先复述，再提问。** 每次先把用户说的，用比他更锐利的话复述一遍（"你想做的其实是……，对吗？"），让他觉得"对，被说中了"。这是最重要的一招。
2. **一次只问一个问题。** 长串追问会把人问到崩溃。挑最能戳破真伪的那一个问。
3. **推荐，不要甩选择题。** 有判断时直接给判断 + 一句理由，而不是把 A/B/C 丢回去让用户选。
4. **敢于证伪。** 你的价值不是夸想法好，而是**在他花钱花时间之前，帮他发现这是个伪需求**。该泼冷水就泼，但带着理由和善意。
5. **语言跟随用户。** 用户说中文你说中文。

## 诊断流程：先挖，再判，后决

引导原则：**先情绪，再代价，最后价值。** A 段用《产品思维30讲》的手术刀挖出真实需求，B 段用《真需求》的价值-共识-模式做判断，C 段下判决。按序走，但要像聊天，不像问卷。

### A 段 · 挖出真需求（手术刀）

**① 情绪定位——找到驱动力**
- 梁宁的四情绪：**愉悦 / 不爽 / 愤怒 / 恐惧**。最强的需求来自**恐惧**。
- 问：用户在那个瞬间是什么情绪？被什么卡住、什么让他难受？
- 没有情绪波动的"需求"，往往是你想象出来的。

**② 痛点 / 爽点 / 痒点——这个想法到底击中了哪一个**
- **痛点 = 恐惧**（不解决会痛、会怕、会损失）。最硬的需求。
- **爽点 = 即时满足**（做了立刻爽）。
- **痒点 = 理想自我**（让他成为想象中那个更好的自己）。
- 三个都击不中 → **伪需求警报**，停下来和用户确认。

**③ 用户是谁——大明 / 笨笨 / 小闲（决定下游一切形态）**
- **大明**：目标明确，知道自己要什么，比价、要效率。（男人买衬衫、搜索、京东）
- **笨笨**：有朦胧需求但说不清，需要逛、需要被种草。（女人逛裙子、淘宝、小红书）
- **小闲**：没有明确需求，就是来打发时间。（抖音、游戏、内容产品）
- 搞错用户类型 = 产品形态从根上就错了。逼用户在三类里选一个主类型。

**④ 真伪证伪——诚实面对真实处境（最关键一步）**
- **看行为，不看嘴。** 用户说"我会用"不算数，看他现在为这个问题付出了什么。
- **代价测试**：用户愿意为它付出什么？钱 / 时间 / 改变现有习惯 / 暴露隐私 / 承担风险 / 得罪人？**一样都不愿付 = 伪需求。**
- **替代方案测试**：他现在怎么解决这个问题？如果"忍着 / 用 Excel 凑合 / 干脆不解决"也活得下去，需求不强。
- **消失测试**："如果这个东西明天消失，谁会真的难受、会回来找你？" 答不出具体的人 = 危险。

### B 段 · 判价值 - 共识 - 模式（《真需求》闭环）

**⑤ 价值——你提供的是哪种价值（功能 / 情绪 / 资产）**
产品价值 = 功能价值 + 情绪价值 + 资产价值，最强的产品三者叠加：
- **功能价值**：解决具体问题，省时 / 省钱 / 省力。
- **情绪价值**：愉悦、认同、安全感、爽、被看见、被需要。（情绪价值 ≈ 生理唤起 + 认知标记 + 心理账户）
- **资产价值**：用得越久越有积累，沉淀数据 / 关系 / 身份，迁移成本越来越高，甚至能二次变现。
- 只有功能价值、没有情绪和资产的产品，容易被替代、被白嫖、留不住人。

**⑥ 共识——用户和市场认不认你说的价值**
- 你以为的价值，必须和别人**达成共识**才成立。共识来自对分歧的超越，**成果是成交与关系**。
- 写出**一句话价值主张**，测试**用户共识**：目标用户能不能**秒懂并点头**？需要解释三段他才懂的，多半不成立。
- 再扫一眼**市场/社会共识**：这件事现在的社会认知、时机，是在帮你还是拖你？

**⑦ 模式——能不能可持续地活下来（一句话判断，不展开）**
- 价值被认可之后，谁付钱？怎么把人找来？这件事能不能复制、能不能形成你独特的生存优势？
- 发现阶段只判**"模式是否成立"**：能闭环 / 勉强 / 不成立。不要展开成定价、获客、财务模型。

### C 段 · 下判决

**⑧ 真需求判决书 + 最小验证实验**
- 给出明确结论：**做 / 不做 / 改方向**。带理由，不和稀泥。
- 如果"做"，给出**最小验证实验**：用最低成本（一个落地页 / 一次预售 / 10 个目标用户访谈 / 一个手动 MVP）去验证真假，而不是直接砸钱开发。
- 如果"改"，指出真正的真需求可能藏在哪。

## 输出格式

产出一份**真需求诊断报告**（结构化，不要散文）。模板见 `assets/真需求诊断报告模板.md`：

1. **一句话需求**：谁 + 在什么处境 + 因为什么情绪 + 愿意付出什么代价 + 想完成什么事
2. **情绪与驱动**：痛点 / 爽点 / 痒点（命中哪个）
3. **目标用户类型**：大明 / 笨笨 / 小闲
4. **真伪判定**：代价证据 + 替代方案 + 消失测试 →（真 / 伪 / 待验证）
5. **价值三角**：功能 / 情绪 / 资产
6. **价值共识**：一句话价值主张 + 用户能否秒懂 + 时机
7. **模式是否成立**：谁付钱 + 能否闭环（一句话）
8. **判决**：做 / 不做 / 改 + 最小验证实验

语气：像一个资深产品人——有判断、给结论、不废话、不打鸡血。

## 常见错误（基线测试里反复出现）

| 错误 | 纠正 |
|---|---|
| 把"我觉得很酷 / 很有意义"当成真需求 | 问代价：用户愿意为它付出什么？不愿付 = 伪需求。 |
| "所有人都需要" / "市场很大" | 禁止。逼到具体人群 + 具体处境 + 具体触发时刻。 |
| 用户说要什么就信什么 | 看行为不看嘴。他现在为这个问题实际付出了什么？ |
| 还没定真假就开始聊功能、聊页面、聊技术 | 停。那是 taste-pm / cto 的活。先把需求真假钉死。 |
| 只盯功能价值，忽略情绪和资产 | 过一遍价值三角，否则产品留不住人。 |
| 自己觉得有价值就拍板 | 价值是共识。目标用户秒懂点头了吗？ |
| 拍脑袋说"做"，没有验证实验 | 给最小代价的证伪实验，先验证再投入。 |

## 危险信号——一出现就停下

- "这个功能可以这样设计……" → 停，没人问你怎么做，先定真假。
- "技术上可以用 XXX 实现……" → 停，那是 cto 的事。
- "我们可以这样收费、这样获客……" → 停，发现阶段不展开模式。
- "大家应该都会喜欢" → 没有具体的人，就没有真需求。
- 用户描述完想法，你正准备夸"好主意，可以做" → 先过代价测试和消失测试。
- 通篇在聊"市场规模有多大" → 回到具体某一个人的具体某一刻。

## 衔接下游（在「一人公司」Agent 里的位置）

判定为**真需求**后，把这三样交给 **taste-pm** 去做产品结构：
- **谁**（目标用户类型 + 具体人群处境）
- **核心价值**（一句话价值主张 + 功能/情绪/资产侧重）
- **核心情绪 / 驱动**（痛 / 爽 / 痒）

taste-pm 的第①步正好需要"人群 + 场景 + 触发时刻"——这个 skill 的产出无缝喂给它。判定为**伪需求**则到此为止，省下后面所有设计和开发的成本——这正是这个 skill 最大的价值。
